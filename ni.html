Technology Stack: The "What" and the "Why"

Each technology was chosen for a specific advantage it brings to the project.

  • Next.js 15 (App Router)
    Advantage: Provides a unified framework for both frontend and backend. API Routes (`/app/api`) eliminate the need for a separate Node.js server, drastically simplifying development and deployment. Server Components allow for efficient data fetching directly where it's needed.
  • Supabase
    Advantage: It's more than just a database; it's a full backend suite. The key advantage is its powerful Row Level Security (RLS) built on PostgreSQL. This lets us define data access rules directly in the database, ensuring users can *only* ever see their own data. This is more secure and scalable than filtering data in application code.
  • Zustand
    Advantage: Simplicity and performance. It avoids the heavy boilerplate of other state management libraries like Redux. Its `persist` middleware is a one-line solution for saving the user's session to local storage, providing a seamless experience if they refresh the page.
  • Google Gemini API (via Direct REST)
    Advantage: Full control. By using direct HTTP requests, we have complete transparency into the request/response cycle. This avoids any potential bugs or limitations of an intermediate SDK and makes the application's core AI logic dependency-free and easy to maintain.
  • Tailwind CSS
    Advantage: Speed and consistency. It allows us to build custom, complex user interfaces rapidly without writing custom CSS files. All styling is co-located with the components, making them easy to understand and modify.

Part 2: The Creative Workflow

Phase 1: Idea Generation

This is where the user's journey begins. The goal is to move from a vague topic to concrete, strategic directions.

User Perspective

The user sees a clean interface with a large text area. They type their presentation topic, for example, "The impact of remote work on company culture." They see a slider to select the number of slides and a single button: "Generate Angles".

End-to-End Flow

  1. User Action: The user clicks the "Generate Angles" button.
  2. Frontend Logic (idea-view.js & zustand-store.js):
    • The `onClick` event handler calls the `generateAngles(topic)` action from the Zustand store.
    • The store immediately sets its `isLoading` state to `true`. This causes a loading spinner to appear on the button, disabling it to prevent multiple clicks.
    • The store uses the browser's `fetch` API to make a POST request to the application's own backend endpoint, `/api/ai`.
    • The body of the request is a JSON object: { action: 'generate_angles', payload: { topic: 'The impact...' } }.
  3. Backend Logic (api/ai/route.js & core/ai.js):
    • The Next.js server receives the request at `route.js`. It parses the JSON body and identifies the `action` as `'generate_angles'`.
    • It then calls the corresponding function: `generateStrategicAngles(topic)` located in the `core/ai.js` file.
  4. AI Integration (core/ai.js):
    • The `generateStrategicAngles` function constructs a highly specific prompt for the Google Gemini API. The prompt instructs the AI to act as a "world-class presentation strategist" and to return a response formatted as a clean JSON object containing an array of three distinct angles.
    • A secure, server-to-server API call is made to Google Gemini.
    • **Resilience:** If the API call fails or the returned JSON is invalid, a `catch` block triggers a fallback mechanism. This fallback logic generates sensible default angles (e.g., "The Productivity Angle," "The Employee Wellness Angle") to ensure the user always receives a valid response.
  5. Result: The backend sends the JSON array of angles back to the frontend. The Zustand store receives this data, sets `isLoading` to `false`, and updates its `presentation.strategicAngles` state. React automatically re-renders the `idea-view.js` component, displaying the new angles as a list of clickable buttons.
Code Spotlight: The core AI logic for this phase is in the generateStrategicAngles function within /src/core/ai.js.

Phase 2: Outline Creation

Here, the chosen strategy is expanded into a full presentation plan, or "Blueprint."

User Perspective

The user reviews the generated angles and clicks on the one that best fits their goal. The view immediately transitions to the "Outline" page, where a loading animation is displayed. After a few moments, a detailed, slide-by-slide outline appears. To the side, there is a chat box labeled "AI Chat" where they can type requests to modify the outline.

End-to-End Flow

  1. User Action: The user clicks on an angle button.
  2. Frontend Logic (idea-view.js & zustand-store.js):
    • The `onClick` handler calls the `chooseAngleAndBuildBlueprint(angle)` action.
    • This action performs an **optimistic UI update**: it immediately sets `activeView` to `'outline'`. This makes the app feel fast, as the UI changes before waiting for the backend.
    • It sets `isLoading` to `true` and sends a new request to `/api/ai` with the action `'generate_blueprint'`.
  3. Backend Logic (api/ai/route.js & core/ai.js):
    • The API route receives the request and calls the `generateBlueprint(topic, angle, slideCount)` function.
  4. AI Integration (core/ai.js):
    • This function uses the most detailed prompt in the application. It instructs the AI to act as a "Creative Director and Brand Designer."
    • It explicitly demands two things:
      1. A complete **Generative Design System (GDS)**: A JSON object containing a color palette, font pairings, and mood keywords.
      2. A slide-by-slide breakdown, emphasizing the use of rich content blocks (like `statistic_highlight` or `pull_quote`) over simple bullet points.
    • The AI generates and returns a single, large JSON object. The backend code then validates and sanitizes this object to ensure it matches the application's data model before sending it to the frontend.
  5. Refinement Loop: If the user types "Make slide 3 more formal" into the chat box in outline-view.js, the `refineBlueprintViaChat` action sends the *entire current blueprint* and the chat message to the backend. The AI is instructed to make the change and return the *full, updated blueprint*, ensuring the frontend state can be replaced atomically and remains consistent.
Code Spotlight: The complex prompt for creating the blueprint is in the generateBlueprint function in /src/core/ai.js. The UI is managed by /src/app/dashboard/outline-view.js.

Phase 3: Deck Rendering & Export

The final phase turns the abstract blueprint into tangible, viewable slides.

User Perspective

After finalizing the outline, the user clicks "Finalize & Generate Presentation." The view switches to the "Deck" page. They see a placeholder animation that looks like code being written. Then, thumbnails of the slides begin appearing on the left, one by one. The main area shows a large preview of the currently selected slide. There are controls to switch between a "Preview" and a "Code" view, and an "Export PPTX" button.

End-to-End Flow

  1. User Action: The user clicks "Finalize & Generate Presentation".
  2. Frontend Logic (zustand-store.js & deck-view.js):
    • The `generateRecipes()` action is called. It switches the `activeView` to `'deck'` and clears any old slide recipes.
    • It initiates a streaming `fetch` request to `/api/ai` with `action: 'generate_recipes_stream'`.
    • The code then enters a `while` loop, reading from the response stream. As each text chunk (representing one slide recipe) arrives, it's parsed and added to the `slideRecipes` array in the store. This incremental update is what creates the "one by one" generation effect.
    • The `deck-view.js` component renders the `slide-renderer.js` component, passing it the current slide's recipe.
  3. Backend Logic & AI Integration (core/ai.js):
    • The `api/ai/route.js` calls the `generateSlideRecipesStream()` async generator function.
    • This function doesn't return all the data at once. It first `yields` the theme data. Then, it loops through the slides in the blueprint. For each slide, it constructs a prompt asking the AI to generate a renderable recipe. A key instruction is for the AI to prefer generating self-contained HTML/CSS/JS (`code` mode) that uses CSS variables for theming.
    • As soon as the AI returns a recipe for a single slide, the function `yields` it, sending it immediately to the client. This process repeats for all slides.
  4. PPTX Export Flow (api/export-pptx/route.js):
    • When the user clicks "Export PPTX", the frontend sends the *final* presentation JSON data to this separate, dedicated API route.
    • This route does **not** call the AI. It uses the `pptxgenjs` library to programmatically construct a PowerPoint file in memory. It iterates through the slide recipes, adding shapes, text boxes, and placeholders to each slide based on the JSON data.
    • The final file is sent back to the browser as a binary buffer, triggering a download.
Code Spotlight: The streaming logic is in the `generateSlideRecipesStream` function in /src/core/ai.js and the corresponding `generateRecipes` action in /src/utils/zustand-store.js. Slide rendering is handled by /src/components/slide-renderer.js.

Part 3: The Developer's Deep Dive

Database Schema Explained

The entire database is defined in a single SQL file for reproducibility. Here's a breakdown of the key parts.

Supabase Row Level Security (RLS) Explained

-- This command enables advanced cryptographic functions, like generating UUIDs.
create extension if not exists pgcrypto;

-- This is a reusable function that we'll attach to tables.
-- It automatically sets the 'updated_at' column to the current time on any change.
create or replace function public.set_updated_at()
returns trigger language plpgsql as $$
begin
  new.updated_at = now();
  return new;
end; $$;

-- This is the main table storing all user presentations.
create table if not exists public.presentations (
  id uuid primary key default gen_random_uuid(), -- A unique ID for each presentation.
  user_id uuid not null, -- The ID of the user who owns this presentation. CRITICAL for security.
  topic text not null default '',
  blueprint jsonb, -- The 'jsonb' type is a binary, indexed format for JSON, making queries faster.
  recipes jsonb,
  status text not null default 'idea', -- Tracks the workflow stage.
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

-- Attaching the trigger to our table.
create trigger trg_presentations_updated_at
  before insert or update on public.presentations
  for each row execute procedure public.set_updated_at();

-- This command turns on Row Level Security for the table. By default, no one can see anything.
alter table public.presentations enable row level security;

-- Now we define a policy. This is a rule that allows access.
-- This policy says: "A user is allowed to SELECT (read) rows from this table
-- ONLY IF the 'user_id' in that row is equal to their own authenticated user ID."
drop policy if exists "presentations_select_own" on public.presentations;
create policy "presentations_select_own" on public.presentations for select
using (user_id = auth.uid());

-- We create similar policies for INSERT, UPDATE, and DELETE.

Frontend Architecture

The frontend is a reactive UI built on a foundation of component architecture and centralized state management.

Zustand Unidirectional Data Flow

  1. A user interacts with a React Component (e.g., clicks a button).
  2. This calls an Action in the Zustand store (e.g., generateAngles()).
  3. The Action performs logic (like fetching from an API) and then updates the central Store.
  4. Because the React Component is subscribed to the store, the state change automatically triggers a re-render of the UI with the new data.

Backend Architecture

The backend is lean and powerful, leveraging serverless functions within Next.js to handle all business logic.

Backend Orchestrator Pattern

The Orchestrator Pattern: Instead of having many small API endpoints, we use a single endpoint: /api/ai/route.js. The frontend sends a request with an action field (e.g., 'generate_blueprint'). The route then acts as a traffic controller, calling the correct function from /core/ai.js based on the action. This keeps the API surface clean and simplifies security.

Prompt Engineering in /core/ai.js

This is where we "program" the AI. The quality of the AI's output is almost entirely dependent on the quality of our prompts. Key strategies include:

  • Persona Setting: We tell the AI who to be (e.g., "You are a world-class presentation strategist...") to get expert-level results.
  • Strict Schema Enforcement: We provide a detailed JSON structure in the prompt and command the AI to follow it exactly.
  • Robust Fallbacks: Every function that calls the AI is wrapped in a try...catch block. If the AI fails, the function returns a sensible, hard-coded default. This makes the application resilient.

Local Setup Guide

Follow these steps to get a complete, working version of Nether AI running on your local machine for development.

  1. Prerequisites: Ensure you have Node.js (version 20 or newer is recommended) and Git installed on your system.
  2. Clone the Repository: Open your terminal or command prompt and run the following command to download the project source code: 
    git clone <repository_url>
cd <project_folder>
  3. Install Dependencies: Run the Node Package Manager (npm) to install all the libraries listed in package.json. 
    npm install
  4. Set Up Your Supabase Project:
    • Go to supabase.com, sign up for a free account, and create a new project.
    • Once your project is ready, navigate to Project Settings (the gear icon) and then to the API tab.
    • Here you will find your Project URL and your anon public key. Keep this page open.
  5. Configure Your Local Environment:
    • In your code editor, create a new file at the root of the project and name it exactly .env.local.
    • Copy and paste the following into this file, replacing the placeholder text with the actual keys from your Supabase project and your Google Gemini API key.
    # .env.local - This file contains your secret keys and is ignored by Git.
NEXT_PUBLIC_SUPABASE_URL=YOUR_SUPABASE_PROJECT_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY=YOUR_SUPABASE_ANON_KEY
GOOGLE_GEMINI_API_KEY=YOUR_GOOGLE_GEMINI_API_KEY
  6. Initialize the Database Schema:
    • Go back to your Supabase project dashboard and navigate to the SQL Editor.
    • Open the SUPABASE_SCHEMA_AND_SQL.txt file from the project's source code. Select and copy its entire contents.
    • Paste the copied SQL into the Supabase SQL Editor and click the "RUN" button. This will create all the tables, security policies, and functions your application needs.
  7. Create the Storage Bucket:
    • In Supabase, navigate to the Storage section from the main menu.
    • Click "New bucket", name it presentation-assets, and make sure the "Public bucket" switch is turned ON. This allows the application to display uploaded images.
  8. Start the Development Server: 
    npm run dev

    The application should now be running. Open your web browser and navigate to http://localhost:3000 to see it in action.
