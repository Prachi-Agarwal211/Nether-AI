Nether AI – Project Documentation (Comprehensive)

Last updated: 2025-08-12

1) Introduction
- Project Overview: Nether AI is a Next.js-based app that helps users generate professional presentation content and design blueprints using AI, store artifacts securely, and export to PPTX. It integrates Supabase for authentication, database, and storage; and uses Google Gemini models for AI generation.
- Audience: Developers onboarding to the project, contributors, and maintainers.
- Key Features:
  • Email/Password + OAuth authentication via Supabase
  • AI-powered generation of strategic angles and slide blueprints
  • Slide rendering in a 12-column grid with rich content blocks and code-based rendering in a sandboxed iframe
  • Autosave and history persistence to Supabase
  • Server-side PPTX export via API

2) Tech Stack
- Framework: Next.js 15 (App Router) – `next: 15.4.5`
- Language: React 19 – `react: 19.1.0`, `react-dom: 19.1.0`
- Styling: Tailwind CSS 3 – `tailwindcss: ^3.4.17`
- State Mgmt: Zustand – `zustand: ^4.5.2`
- Backend-as-a-Service: Supabase – `@supabase/supabase-js: ^2.43.4`, `@supabase/ssr: ^0.6.1`
- AI: Google Gemini (via direct REST fetch). Some AI SDK packages are present but not actively used in current code.
- PPTX: `pptxgenjs: ^4.0.1` (server-side export route)
- Animations/UI: `framer-motion`, `react-icons`, `react-hot-toast`, `lucide-react`
- 3D/Visual (optional components): `three`, `@react-three/fiber`, `@react-three/drei`

4) Why each file/folder exists
- `src/app/page.js` – Auth landing. Implements sign in/up/reset and Google OAuth using Supabase. Redirects authenticated users to `/dashboard`.
- `src/app/layout.js` – Root layout, fonts, global CSS, `react-hot-toast`, and lazy `VantaBackground`.
- `src/app/api/ai/route.js` – Central API route to invoke AI tasks:
  • `generate_angles` → `core/ai.generateStrategicAngles()`
  • `generate_blueprint` → `core/ai.generateBlueprint()`
  • `refine_blueprint` → `core/ai.refineBlueprintViaChat()`
  • `generate_recipes` / `generate_recipes_stream` → slide recipe generation or streaming (iterator) if implemented in `core/ai.js`
- `src/app/api/export-pptx/route.js` – Receives presentation plus slide recipes and returns a PPTX file generated with `pptxgenjs`.
- `src/components/slide-renderer.js` – Renders slide recipes in a twelve-column CSS grid with multiple element types (Title, BulletedList, Paragraph, Quote, Stat, Callout, Image, Diagram, Table) and a sandboxed iframe mode if a recipe contains AI-provided `code.html/css/js`.
- `src/core/ai.js` –
  • `callGoogleGemini()` – Direct REST call to Gemini (`v1beta/models/${model}:generateContent`). Uses `GOOGLE_GEMINI_API_KEY` (server) or `NEXT_PUBLIC_GOOGLE_GEMINI_API_KEY` as fallback.
  • `generateStrategicAngles(topic)` – Generates 2–3 presentation angles with validation and robust fallbacks by topic domain.
  • `generateBlueprint(topic, angle, slideCount)` – Produces a full presentation blueprint with theme GDS and slide blocks; contains deterministic fallback.
  • `refineBlueprintViaChat(blueprint, chatHistory)` – Refines blueprint with constraints and safety checks.
  • `generateBlueprintStreaming(...)` – Streaming scaffolding (continues in file) for progressive UX.
- `src/utils/zustand-store.js` –
  • Holds `AppState` (active view, presentation model, chat/history)
  • Supabase persistence: `loadHistory()`, `savePresentation()`
  • AI actions: `generateAngles()`, `chooseAngleAndBuildBlueprint()`
  • Media upload to Supabase Storage: `uploadAttachment()` into `presentation-assets`
  • Debounced autosave `_scheduleAutosave()`
- `src/utils/supabase-client.js` – Creates browser client using `createBrowserClient(NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY)`.
- `SUPABASE_SCHEMA_AND_SQL.txt` – End-to-end SQL for tables: `presentations`, `assets`, `profiles`; helper triggers, RLS policies; storage bucket configuration steps; Google sign-in steps; verification and optional reset SQL.
- `next.config.mjs` – Adds security headers for `/api/*`: `X-Content-Type-Options`, `X-Frame-Options`, `Cache-Control: no-store`.
- `tailwind.config.js` – Tailwind paths, fonts (Geist), custom colors and animations.
- `eslint.config.mjs` – ESLint setup for Next.js best practices.

7) Supabase Setup (Phase-by-Phase)
Phase 1 – Project and Auth:
- Create a Supabase project (Dashboard). Get `Project URL` and `anon key` for `.env.local`.
- Authentication:
  • Enable Email/Password (default)
  • Optional: Enable Google provider. Configure OAuth in Google Cloud Console and paste secrets into Supabase. Add dev Site URL (http://localhost:3000) in Supabase Auth settings.

Phase 2 – Database Schema and Policies:
- Open Supabase SQL Editor and run all statements in `SUPABASE_SCHEMA_AND_SQL.txt` under “Creation SQL”. This will:
  • Create tables: `presentations`, `assets`, `profiles`
  • Create helper triggers: `set_updated_at`, `set_user_id`, `set_profile_ids`
  • Enable RLS and add policies for per-user access
  • Add helpful indexes
- Run the “Verification SQL” queries in the same file to confirm objects/policies exist.

Phase 3 – Storage (presentation-assets):
- Create storage bucket `presentation-assets` (public ON) via Dashboard: Storage > Buckets > New bucket.
- Using the UI, add policies for public read, authenticated insert, and optional owner update/delete as described in `SUPABASE_SCHEMA_AND_SQL.txt`.

Phase 4 – Environment and App bootstrap:
- Set `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local` and in your hosting provider (Vercel) project env.
- Set `GOOGLE_GEMINI_API_KEY` in environment for server-side AI requests.

8) Frontend Architecture and Flow
Architecture:
- App Router with pages in `src/app/`
- Client-first auth page at `src/app/page.js`, with animations via `framer-motion`
- Visuals via `VantaBackground` and Tailwind design system
- Components in `src/components/`, including `slide-renderer.js` for rendering recipes (grid mode and code iframe mode)

State Management (Zustand) – `src/utils/zustand-store.js`:
- Global state: `activeView` ('idea' | 'outline' | 'deck'), `presentation` model (topic, angles, chosenAngle, slideCount, blueprint, slideRecipes, themeRuntime, activeSlideIndex), history, chatHistory
- Key actions:
  • `generateAngles(topic)` → calls `/api/ai` with action `generate_angles`
  • `chooseAngleAndBuildBlueprint(angle)` → calls `/api/ai` with action `generate_blueprint`
  • `savePresentation()` and `_scheduleAutosave()` → persist to Supabase `presentations`
  • `loadHistory()` → loads user’s presentations list
  • `uploadAttachment(file, slideId)` → uploads to Supabase Storage and updates blueprint attachments

Views (high-level):
- Idea View (`src/app/dashboard/idea-view.js`): Topic input, request angles from AI, select one.
- Outline View (`src/app/dashboard/outline-view.js`): Displays/edit blueprint; may call `refine_blueprint`.
- Deck View (`src/app/dashboard/deck-view.js`): Renders slides using `slide-renderer.js`, preview grid, and optionally triggers export.

Rendering – `slide-renderer.js`:
- Two modes:
  • Element Grid: interpret `recipe.elements` mapped to a 12-column CSS grid (Title, BulletedList, Paragraph, Quote, Stat, Table, Image, Callout, Diagram)
  • Code Iframe: if `recipe.code.{html,css,js}` exists, builds a sandboxed iframe `srcDoc` with CSS vars derived from theme runtime; bridges console errors to parent for debugging
- Backgrounds:
  • Solid color or generated backgrounds (e.g., gradient or noise canvas) with cleanup

9) Backend/API Flow
AI Route – `src/app/api/ai/route.js`:
- Input: `{ action, payload }` via POST
- Requires env: `GOOGLE_GEMINI_API_KEY` (server) or `NEXT_PUBLIC_GOOGLE_GEMINI_API_KEY` fallback
- Actions:
  • `generate_angles` → returns `{ angles: [...] }`
  • `generate_blueprint` → returns the blueprint JSON with theme and slides
  • `refine_blueprint` → returns updated blueprint
  • `generate_recipes` / `generate_recipes_stream` → slide recipes or streaming payloads (if implemented)
- Streaming helper composes SSE-style `text/event-stream`

PPTX Export – `src/app/api/export-pptx/route.js`:
- Input: `{ presentation, recipes, slides?, theme_runtime?, theme_gds? }`
- Builds PPTX with `pptxgenjs`:
  • Adds slides; maps grid positions to coordinates; renders Title, BulletedList, Paragraph, Quote, Stat, Table, Image placeholder, Callout
  • Sets presentation meta and slide numbers
- Output: `application/vnd.openxmlformats-officedocument.presentationml.presentation` buffer with Content-Disposition

Supabase Interactions:
- Client: `supabase-client.js` using anon key; `zustand-store.js` performs `from('presentations').upsert(...)`, `select(...)` history, and Storage uploads to `presentation-assets`
- Service Role: not currently used; do not expose service role key client-side

- 401/Forbidden from Supabase:
  • Ensure `.env.local` values are correct
  • Confirm RLS policies exist and user is authenticated
- AI route errors:
  • Verify `GOOGLE_GEMINI_API_KEY` is set in the server environment
  • Check server logs for `[AI_ROUTE_ERROR]`
- PPTX export fails:
  • Ensure request body matches route expectations and recipes include supported elements
  • Investigate server logs for `PPTX Export Error`
- Storage upload fails:
  • Verify `presentation-assets` bucket exists and policies allow auth users to insert

17) Data Models and Relationships
Tables (see `SUPABASE_SCHEMA_AND_SQL.txt` for full DDL and policies):

- presentations
  • id (uuid, PK) – unique presentation id
  • user_id (uuid, not null) – owner; auto-set via trigger to `auth.uid()`
  • topic (text, not null, default '') – user-entered topic
  • chosen_angle (jsonb) – selected strategic angle metadata
  • slide_count (int, default 10) – requested number of slides
  • blueprint (jsonb) – source-of-truth presentation structure (theme, slides, blocks)
  • recipes (jsonb) – render-ready slide recipes (optional, may be produced later)
  • status (text, default 'idea') – workflow stage: idea|outline|deck|draft
  • created_at/updated_at (timestamptz) – timestamps; updated via trigger
  Indexes: `(user_id, updated_at desc)`
  RLS: select/insert/update/delete only when `user_id = auth.uid()`

- assets (optional Phase 1 table)
  • asset_id (uuid, PK) – asset id
  • user_id (uuid, not null) – owner; auto-set via trigger
  • presentation_id (uuid, FK -> presentations.id, on delete cascade)
  • type (text) – 'image' | 'doc' | 'data'
  • file_name (text), content_type (text), size_bytes (int)
  • storage_path (text) – path in Storage bucket (e.g., `presentation-id/slide-id/file`)
  • public_url (text) – public URL for display
  • created_at (timestamptz)
  RLS: owner-only policies

- profiles
  • id (uuid, PK) – mirrors `auth.users.id`
  • user_id (uuid) – convenience for RLS/indexing
  • email, first_name, last_name, username (unique), phone, date_of_birth
  • created_at/updated_at (timestamptz)
  RLS: row access restricted to self

Storage
- Bucket: `presentation-assets` (public ON)
- Policies: public read, authenticated upload, optional owner update/delete (configure via UI)

18) Happy Path – User Journey Narrative
1. Onboarding
   - User visits `/` (`src/app/page.js`), signs up/signs in (email/password or Google OAuth via Supabase).
   - On success, Next.js redirects to `/dashboard`.
2. Idea Generation
   - In idea view, user enters a topic and clicks “Generate Angles”.
   - Store action `generateAngles(topic)` POSTs to `/api/ai` with `action: 'generate_angles'`.
   - API calls Gemini; frontend displays 2–3 angles.
3. Blueprint Creation
   - User selects an angle -> `chooseAngleAndBuildBlueprint(angle)` calls `/api/ai` with `action: 'generate_blueprint'`.
   - Store sets `activeView = 'outline'` and updates `presentation.blueprint` with the result.
   - Background `savePresentation()` upserts to Supabase `presentations` table.
4. Deck Interaction & Refinement
   - User opens deck view; `slide-renderer.js` renders recipes or blueprint-derived elements.
   - Edits and attachments trigger `_scheduleAutosave()` -> `savePresentation()` to persist.
   - `uploadAttachment(file, slideId)` uploads to Storage and updates blueprint attachments.
5. Export
   - User clicks Export; frontend POSTs `{ presentation, recipes }` to `/api/export-pptx`.
   - Server generates PPTX with `pptxgenjs`; browser downloads the file.

19) Error Handling & Logging Strategy
Client-Side
- Use `react-hot-toast` for non-blocking UX feedback (success/info/warn).
- For actionable failures, show a clear error toast/message; `console.error('[CLIENT_ERROR]', err)` for debugging.
- Avoid blocking the UI unless absolutely necessary; keep edits non-destructive.

API/Server-Side
- Wrap handlers in try/catch; return standardized JSON errors: `{ error: string }` with appropriate HTTP status.
- Add searchable log prefixes:
  • AI route: `console.error('[AI_ROUTE_ERROR]', err)` in `src/app/api/ai/route.js`.
  • PPTX export: `console.error('PPTX Export Error:', error)` in `src/app/api/export-pptx/route.js`.

Structured Logging (Roadmap)
- Integrate Sentry/Logtail/Vercel Logging for production error aggregation and alerting.
- Capture request IDs and user IDs (when available) for correlation.
